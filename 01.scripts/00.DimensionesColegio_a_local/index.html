    Documentación del Script SQL para el Esquema Dimensiones Colegio¶

    Introducción¶

    Este script SQL pertenece a un sistema de manejo de datos en un Data Warehouse, específicamente en el esquema Colegio. Su propósito es eliminar restricciones de claves foráneas en las tablas del esquema, y luego realizar configuraciones de integridad referencial mediante la definición de claves primarias. A continuación, se detallan las secciones del código.

    Descripción del Código¶

    1. Contexto Inicial¶

    El script comienza seleccionando la base de datos DWH_COMFENALCO y establece configuraciones iniciales como ANSI_NULLS y QUOTED_IDENTIFIER para asegurar el comportamiento estándar de SQL Server.

    USE DWH_COMFENALCO;
GO
SET ANSI_NULLS ON;
GO
SET QUOTED_IDENTIFIER ON;
GO
    • USE DWH_COMFENALCO: Cambia el contexto de ejecución a la base de datos DWH_COMFENALCO.
    • SET ANSI_NULLS ON: Habilita el tratamiento estándar de valores NULL en operaciones comparativas.
    • SET QUOTED_IDENTIFIER ON: Permite usar nombres de identificadores entre comillas dobles.

    2. Eliminar Restricciones de Claves Foráneas¶

    Se utiliza un procedimiento dinámico para eliminar todas las restricciones de claves foráneas asociadas al esquema Colegio.

    DECLARE @sql NVARCHAR(MAX) = '';
SELECT @sql += 'ALTER TABLE ' + QUOTENAME(s.name) + '.' + QUOTENAME(t.name) 
    + ' DROP CONSTRAINT ' + QUOTENAME(f.name) + '; ' 
FROM sys.foreign_keys f
INNER JOIN sys.tables t ON f.parent_object_id = t.object_id
INNER JOIN sys.schemas s ON t.schema_id = s.schema_id
WHERE s.name = 'Colegio';
EXEC sp_executesql @sql;

    Desglose:¶

    1. sys.foreign_keys: Obtiene información de todas las claves foráneas en las tablas.
    2. sys.tables y sys.schemas: Se utiliza para vincular las tablas a sus esquemas.
    3. Construcción dinámica: El código genera comandos ALTER TABLE para eliminar cada restricción.
    4. EXEC sp_executesql: Ejecuta los comandos generados dinámicamente.

    Motivación: Esto es útil para realizar cambios en las estructuras de las tablas sin conflictos con dependencias existentes.

    3. Creación de Tablas (Comentarios)¶

    Las tablas DIM_CURSO, DIM_GRADO y DIM_POBLACION_MATRICULA están definidas en comentarios como referencia.

    Definición de la tablas¶

    CREATE TABLE [Colegio].[DIM_CURSO](
    [ID_CURSO] [int] NOT NULL,
    [DESC_CURSO] [nvarchar](100) NULL,
    [FECHA_CREACION] [date] NULL,
    [ESTADO_REGISTRO] [nvarchar](20) NULL
) ON [PRIMARY]

CREATE TABLE [Colegio].[DIM_GRADO](
    [ID_GRADO] [int] IDENTITY(1,1) NOT NULL,
    [DESC_GRADO] [nvarchar](100) NULL,
    [FECHA_CREACION] [date] NULL,
    [ESTADO_REGISTRO] [nvarchar](20) NULL
) ON [PRIMARY]

CREATE TABLE [Colegio].[DIM_POBLACION_MATRICULA](
    [ID_POBLACION_MATRICULA] [int] IDENTITY(1,1) NOT NULL,
    [PARTNER] [nvarchar](10) NOT NULL,
    [TIPO_DOCUMENTO] [nvarchar](4) NULL,
    [DOCUMENTO] [nvarchar](20) NULL,
    [NOMBRE_COMPLETO] [nvarchar](4000) NULL,
    [GENERO] [varchar](20) NULL,
    [DIRECCION] [nvarchar](300) NULL,
    [TELEFONO] [nvarchar](30) NULL,
    [CORREO] [nvarchar](250) NULL,
    [FECHA_NACIMIENTO] [int] NULL
) ON [PRIMARY]

    Campos de la Tabla DIM_CURSO¶

    • ID_CURSO: Identificador único de la tabla DIM_CURSO (clave primaria).
    • DESC_CURSO: Descripción del curso en la tabla DIM_CURSO.
    • FECHA_CREACION: Fecha en que se creó el curso en la tabla DIM_CURSO.
    • ESTADO_REGISTRO: Indica si el registro está activo o inactivo en la tabla DIM_CURSO.

    Campos de la Tabla DIM_GRADO¶

    • ID_GRADO: Identificador único de la tabla DIM_GRADO (clave primaria).
    • DESC_GRADO: Descripción del grado en la tabla DIM_GRADO.
    • FECHA_CREACION: Fecha en que se creó el grado en la tabla DIM_GRADO.
    • ESTADO_REGISTRO: Indica si el registro está activo o inactivo en la tabla DIM_GRADO.

    Campos de la Tabla DIM_POBLACION_MATRICULA¶

    • ID_POBLACION_MATRICULA: Identificador único de la tabla DIM_POBLACION_MATRICULA (clave primaria).
    • PARTNER: Identificador del socio en la tabla DIM_POBLACION_MATRICULA.
    • TIPO_DOCUMENTO: Tipo de documento en la tabla DIM_POBLACION_MATRICULA.
    • DOCUMENTO: Número de documento en la tabla DIM_POBLACION_MATRICULA.
    • NOMBRE_COMPLETO: Nombre completo del individuo en la tabla DIM_POBLACION_MATRICULA.
    • GENERO: Género del individuo en la tabla DIM_POBLACION_MATRICULA.
    • DIRECCION: Dirección del individuo en la tabla DIM_POBLACION_MATRICULA.
    • TELEFONO: Teléfono del individuo en la tabla DIM_POBLACION_MATRICULA.
    • CORREO: Correo electrónico del individuo en la tabla DIM_POBLACION_MATRICULA.
    • FECHA_NACIMIENTO: Fecha de nacimiento del individuo en la tabla DIM_POBLACION_MATRICULA.

    4. Definición de Claves Primarias¶

    El script define claves primarias para las tablas DIM_POBLACION_MATRICULA, DIM_CURSO y DIM_GRADO.

    Claves primarias¶

    ALTER TABLE [Colegio].[DIM_POBLACION_MATRICULA]  WITH NOCHECK ADD  CONSTRAINT [PK_DIM_POBLACION_MATRICULA] PRIMARY KEY CLUSTERED ([ID_POBLACION_MATRICULA])

ALTER TABLE [Colegio].[DIM_CURSO]  WITH NOCHECK ADD  CONSTRAINT [PK_DIM_CURSO] PRIMARY KEY CLUSTERED ([ID_CURSO])

ALTER TABLE [Colegio].[DIM_GRADO]  WITH NOCHECK ADD  CONSTRAINT [PK_DIM_GRADO] PRIMARY KEY CLUSTERED ([ID_GRADO])

    Detalles:¶

    • WITH NOCHECK: No verifica los datos existentes en la tabla antes de aplicar la restricción.
    • PRIMARY KEY CLUSTERED: Define una clave primaria con un índice clústerado basado en ID_POBLACION_MATRICULA.

    Claves Primarias Definidas:¶

    Tabla Clave Primaria
    DIM_POBLACION_MATRICULA ID_POBLACION_MATRICULA
    DIM_CURSO ID_CURSO
    DIM_GRADO ID_GRADO

    Buenas Prácticas Implementadas¶

    1. Uso de Restricciones de Integridad:

      • Claves primarias garantizan la unicidad de registros y optimizan las consultas.

    2. Automatización en la Eliminación de Restricciones:

      • Permite realizar modificaciones estructurales masivas sin errores manuales.

    3. Comentarios:

      • Las definiciones de tablas están comentadas como referencia.

    4. Configuraciones Iniciales:

      • Se asegura un comportamiento estándar al trabajar con valores NULL y nombres de columnas.

    5. Modularidad y Dinamismo:

      • El script está diseñado para ser modular y dinámico, facilitando modificaciones estructurales complejas.

    6. Uso de Procedimientos Dinámicos:

      • La construcción dinámica de comandos SQL permite una mayor flexibilidad y adaptabilidad.

    7. Definición Clara de Esquemas:

      • Las tablas y sus relaciones están claramente definidas, lo que mejora la comprensión y mantenimiento del código.

    8. Garantía de Integridad Referencial:

      • La definición de claves primarias y la eliminación controlada de restricciones aseguran la integridad referencial del esquema.
